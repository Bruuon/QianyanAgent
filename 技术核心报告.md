# QianyanAgent 智能体系统技术核心报告

---

## 一、项目概述与开源情况

### 1.1 项目概述

QianyanAgent 是一个基于 Spring Boot 3.5.9 与 LangChain4j 1.1.0 构建的后端技术面试智能问答系统，以 Alibaba Qwen 系列模型（qwen-max、text-embedding-v4、qwen3-rerank）为底座，集成了 RAG 私有知识库检索、多 Agent 协同编排、分布式会话记忆管理、MCP 协议工具调用及全链路可观测性等能力。

系统面向两类核心用户场景：(1) 个人学习者可将私有笔记与面试题集导入向量知识库，获得基于自建资料的定制化回答；(2) 企业可通过本地化部署方案在数据不出域的前提下，让员工以自然语言查询内部 Wiki、SOP 手册、技术文档等私密信息。项目采用 PostgreSQL + PgVector 作为向量存储、Redis 作为分布式会话记忆后端，支持 Docker 容器化部署与 Prometheus + Grafana 运维监控。

### 1.2 项目开源情况

本项目已以 **Apache 2.0** 协议在 GitHub 平台开源。

- **仓库地址**：[https://github.com/Bruuon/QianyanAgent](https://github.com/Bruuon/QianyanAgent)
- **开源协议**：Apache License 2.0

项目技术栈完全基于开源组件构建，不依赖商业闭源依赖，开发者可自由使用、修改与分发。

## 三、项目需求/设计/测试/部署

### 3.1 项目需求

#### 3.1.1 业务场景分析

本系统的定位是"提供个性化知识库服务的智能对话代理"。当前，市面上的主流大模型服务商已先后推出了具备长期记忆与工具调用能力的 Agent 平台。然而，这些平台的共性局限在于：知识库层面仅支持平台预设的通用知识体系，并未向终端用户开放自定义私有知识库的检索增强生成能力。用户无法将自己的学习笔记、技术文档、项目规范等私有知识注入大模型的回答管道。本项目正是针对这一产业缺口，构建一套完整的个性化 RAG 知识库引擎，使用户能够自主定义、管理、检索私有知识，并与大模型的长短期记忆能力配合使用。

**场景一：高度个性化的个人学习需求。** 在技术面试备考、学术研究等知识密集型学习活动中，学习者面临的主要困难不是信息太少，而是信息太多且组织混乱。每个人的知识背景不同——哪些概念已经掌握、哪些还一知半解、喜欢什么样的解释方式——这些差异不可能被通用大模型的一次性回答所照顾到。本系统允许学习者把个人整理的面试题集、技术笔记、课程讲义等私有文档导入向量知识库，此后提问时，系统会从用户自己的资料中检索相关内容，经重排序后作为上下文注入 Prompt，让模型的回答建立在用户自己提供的材料之上，而不是仅靠模型训练时见过的参数化知识。这样做的好处是，同一个问题（比如"Redis 分布式锁怎么实现"），不同用户上传的资料侧重不同，得到的回答侧重点也会不同。

**场景二：企业私密信息的合规问答。** 企业内部有大量高价值信息以非结构化形式散落在 Wiki 文档、SOP 手册、技术规范、项目复盘报告等地方，查找起来费时费力。更大的问题是，企业的源代码、架构设计文档、客户数据、商业策略等绝不能通过公有云大模型 API 传到外部服务器，否则会引发严重的数据合规问题。本系统的 RAG 引擎可以走本地部署路线来解决这个矛盾：企业的知识资产以向量形式存在自建的 PgVector 数据库里，Embedding 模型也在本地运行，全程数据不出企业内网。员工提业务问题时，系统在本地向量库完成检索和重排序，只把检索到的几个文档片段和问题一起拼成 Prompt 发给大模型，不会把整个知识库暴露出去。再加上输入/输出的安全拦截机制，企业可以在用大模型的同时，把信息泄露风险控制在一个可审计的范围内。

#### 3.1.2 功能性需求

**(F1) 多模式对话能力。** 系统须提供三种对话入口：同步阻塞式对话（`POST /api/chat`）、基于 Reactive Stream 的服务端推送流式对话（`POST /api/streamChat`，采用 Project Reactor `Flux<String>` 实现 Token 级实时推送），以及多 Agent 协同对话（`POST /api/multiAgentChat`），后者通过编排层协调知识检索 Agent 与推理 Agent 协同产出最终回复。

**(F2) 检索增强生成（RAG）问答。** 系统须集成私有知识库，对用户提问进行语义检索，将检索到的相关知识片段作为上下文注入大模型 Prompt，以此约束模型输出范围，确保回答基于权威知识源而非模型参数记忆。检索管道包含查询预处理、向量粗排与交叉编码器精排三个阶段。

**(F3) 工具调用（Tool Use / Function Calling）。** 系统须支持大模型自动选择并调用外部工具完成特定任务，包括：查询当前时间（`TimeTool`）、发送电子邮件（`EmailTool`）、动态向知识库添加知识点（`RagTool.addKnowledgeToRag`），以及通过 MCP（Model Context Protocol）协议接入的联网搜索与时间查询等标准化外部工具。

**(F4) 知识库动态更新。** 系统须允许用户通过自然语言指令将新的问答对实时写入知识库，由系统自动完成文档持久化、向量化及向量数据库同步，无需人工干预或服务重启。

**(F5) 安全合规拦截。** 系统须在输入层对用户消息进行敏感词检测与 Prompt 注入防御，一旦命中违禁词即拒绝请求并返回拦截提示，确保生成内容符合合规要求。

#### 3.1.3 非功能性需求

**(NF1) 高并发下的会话状态一致性。** 在多实例部署场景下，同一会话的多次请求可能被路由至不同服务节点，系统必须保证会话记忆的读写一致性。为此引入基于 Redis 的分布式锁机制，确保压缩操作与记忆更新的原子性。

**(NF2) 大模型调用的可观测性。** 所有 LLM 调用须经过统一的监控监听器采集关键指标，包括：请求延迟、Token 消耗量（输入/输出/总计）、首字响应时间（TTFT, Time-to-First-Token）、生成耗时与总耗时。上述指标通过 Micrometer 暴露至 Prometheus，支持 Grafana 可视化。

**(NF3) Token 成本控制与长对话防溢出。** 大模型 API 按 Token 计费，且存在上下文窗口上限。系统须实现自动化记忆压缩策略，在消息累积至阈值时触发历史对话摘要化处理，将长对话的 Token 占用量控制在安全边界内，同时保留关键上下文信息。

**(NF4) 环境隔离与配置安全。** 敏感凭证（API Key、数据库密码）须与代码分离，通过 Spring Profiles 机制实现本地开发环境与生产环境的配置隔离。

### 3.2 项目设计

#### 3.2.1 总体架构

系统采用经典的四层分层架构，自顶向下依次为：接入层（Controller）→ 安全防护层（Guardrail）→ 核心服务层（AiServices）→ 基础设施层（LLM / Memory / RAG / Tools）。各层职责明确，层间通过接口契约解耦。

#### 3.2.2 编排与路由机制（Orchestrator 层）

系统通过 `SimpleOrchestrator`（`orchestrator/SimpleOrchestrator.java`）实现基于策略模式的多 Agent 路由。`Agent` 接口（`agent/Agent.java`）定义 `execute(Long sessionId, String input)` 与 `getAgentName()` 两个核心行为，其实现类分别为：

- **`KnowledgeAgent`**：封装 `RagTool` 引用，负责将用户查询送入 RAG 管道并返回检索结果，异常时返回空字符串保证流程不中断。
- **`ReasoningAgent`**：封装 `AiChat` 接口（由 LangChain4j `AiServices` 动态代理生成），负责调用大模型完成记忆加载、RAG 上下文注入、工具调用及最终回复生成。

路由决策采用轻量级关键词匹配：`SimpleOrchestrator` 维护 `["查询", "了解", "什么是", "介绍", "解释", "说明"]` 静态关键词表，用户输入命中任一关键词则判定为知识查询意图，先由 `KnowledgeAgent` 执行 RAG 检索，将结果格式化为 `"参考知识：\n" + knowledgeResult + "\n\n用户问题：" + userInput` 增强输入；未命中则直接使用原始输入。最终统一调用 `ReasoningAgent` 生成回复。这个设计让 KnowledgeAgent 和 ReasoningAgent 各管一块——一个负责找资料，一个负责整合资料并生成回复——两者通过增强字符串协作，未来要加新的专业 Agent 只需新增实现类并在编排器里加条路由规则即可。

#### 3.2.3 检索增强生成管道（RAG 层）

RAG 层采用"查询预处理 → 向量粗排 → 交叉编码器精排"的三阶段管道架构，核心实现在 `rag/` 包下。

**文档预处理。** `RecursiveDocumentSplitter`（`rag/RecursiveDocumentSplitter.java`）针对中文场景设计了层级分隔符策略（`"。" → "！" → "？" → "\n\n" → "\n" → " " → 字符级截断`），采用递归降级切分：优先以当前层分隔符切分，超出 `maxChunkSize=800` 字符时降级至下层分隔符，最终兜底为字符级硬截断。`chunkOverlap=200` 字符保证相邻片段语义重叠。摄入管道由 `RagConfig` 装配，处理链为：`RecursiveDocumentSplitter → TextSegmentTransformer（文件名元数据注入） → EmbeddingModel（text-embedding-v4, 1024维） → PgVector`。

**查询预处理。** `QueryPreprocessor` 对用户输入执行标点清除、27 个中文停用词过滤及空白规范化，提升查询向量的语义区分度。

**两阶段检索（`ReRankingContentRetriever`）。** 阶段一用 `EmbeddingStoreContentRetriever` 在 PgVector 中做余弦相似度检索，配置 `maxResults=30`、`minScore=0.55`，目标是尽量多召回一些候选，不要漏掉可能相关的文档。阶段二把粗排候选送给 `QwenRerankClient`，调用阿里云 DashScope 的 `qwen3-rerank` 交叉编码器模型做深度语义重排序。和向量检索用的双塔模型不同，交叉编码器会让查询和文档在 Transformer 内部做全自注意力交互，能捕捉到否定句式、术语对齐这类更细的语义关系。最后按 `relevance_score` 降序截取 `finalTopN=5` 条结果。整条链路有四层降级容错：粗排无结果返回空、候选数不够跳过精排、Rerank API 异常降级为向量检索 Top-5、无效索引过滤后兜底。

**知识库热重载。** `RagAutoReloadJob` 以 `@Scheduled(fixedRate = 300000)`（每 5 分钟）基于文件时间戳增量扫描变更文档自动向量化；`RagTool.addKnowledgeToRag` 提供实时写入通道，先追加物理文件再同步向量库。

#### 3.2.4 分布式会话记忆策略（Memory 层）

`CompressibleChatMemory`（`memory/CompressibleChatMemory.java`）实现 LangChain4j 的 `ChatMemory` 接口，在一个类里集成了分布式持久化、并发安全控制和自适应 Token 压缩三项功能。

**存储架构。** 底层基于 `RedisChatMemoryStore`，以 `MemoryId`（即 `sessionId`）为键将消息列表持久化至 Redis，TTL 为 3600 秒，保证多实例部署下会话上下文可恢复。

**分布式锁机制。** `add(ChatMessage)` 方法通过 Redis `SETNX` 命令获取键 `chat:memory:lock:{id}` 的分布式锁（过期 5 秒，重试 3 次，间隔 100ms），防止并发写导致消息丢失。解锁用 Lua 脚本 `if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end` 保证原子性，避免误删别人的锁。重试耗尽后降级为非原子写入——极端竞争下宁可承担极小概率的消息丢失，也不阻塞用户请求。

**Token 感知的自适应压缩。** `TokenCountChatMemoryCompressor` 采用 HuggingFace Tokenizers 加载 Qwen 本地词表（`qwen/tokenizer.json`）进行精确 Token 计数，避免中英文编码估算偏差。压缩触发条件为双阈值逻辑或：`messages.size() > 20` 或 `currentTokenCount > 6000`。压缩采用滑动窗口策略——保留最近 10 条消息（5 轮对话），更早消息压缩为一条包含前 3 条历史消息摘要的 `SystemMessage`，最终结构为 `[1条摘要] + [10条最近消息]`。具备三层降级保护：压缩异常时保留最近 20 条消息；近期消息 Token 超 `recentTokenLimit=2000` 时告警但仍保留；原子更新（Redis `multi/exec` 事务）失败时降级为普通写入。

#### 3.2.5 工具调用与安全防护（Tool & Guardrail 层）

**MCP 协议集成。** `McpToolConfig` 通过 MCP 协议接入两类外部工具：联网搜索客户端使用 `HttpMcpTransport`（HTTP SSE 协议连接智谱 BigModel Web Search 端点），时间查询客户端使用 `StdioMcpTransport`（本地 `uvx` 启动 `mcp-server-time` 进程，时区 `Asia/Shanghai`）。两者统一注册至 `McpToolProvider`，由 `AiChatService` 注入构建器。大模型可自主选择调用，LangChain4j 自动处理 MCP 协议编解码。

**本地工具。** 三个 `@Tool` 注解工具注册于 `AiChatService`：`TimeTool`（获取上海时区时间）、`EmailTool`（基于 Spring Mail 发送 SMTP 邮件，参数由大模型从用户输入自动提取）、`RagTool`（提供 `addKnowledgeToRag` 的对话式知识注入及 `retrieve` 的主动检索能力）。

**输入安全防护。** `SafeInputGuardrail` 实现 `InputGuardrail` 接口，通过 `@InputGuardrails` 注解绑定于 `AiChat` 接口。`validate()` 方法调用 `SensitiveWordBs`（houbb/sensitive-word 库）进行敏感词匹配，命中时返回 `fatal()` 触发 `InputGuardrailException`，由 `GlobalExceptionHandler` 统一处理。`SensitiveWordConfig` 采用系统默认加自定义扩展的双层词库：`CustomWordAllow` / `CustomWordDeny` 分别从类路径文件加载白名单与黑名单词汇。匹配策略关闭数字/邮箱/URL 校验，避免对技术问答中的代码片段误拦截。

### 3.3 项目测试

#### 3.3.1 测试方法论概述

本项目的测试与优化是同步推进的。在全流程的优化过程中，我们没有单凭主观感受或社区评价来选择模型与算法，而是以模型效果的实际量化指标为准绳，在相同测试条件下对比各层次、各算法之间的效果差异，据此决定最终方案。系统的每一次变更（向量化模型替换、查询改写策略调整、文档切分参数修改、Prompt 模板重构），都以可量化后的检索质量指标作为判断依据，确保优化方向由数据验证而非直觉来支撑。

#### 3.3.2 测试基础设施与数据链路

测试体系由三层组件串联构成：

**第一层——批量请求注入。** `scripts/test_agent.py` 负责构造并发送测试题库。50 道题按验证目标分三类：(1) 边界与安全防护测试（10 题），含 Prompt 注入攻击、敏感词绕过、越狱指令等对抗样本，验证 Guardrail 是否扛得住；(2) 域外泛知识测试（20 题），问题范围刻意超出 Redis 知识库的覆盖（如历史、物理、艺术等），考察系统能不能正确识别知识边界并拒绝回答；(3) RAG 核心知识库测试（20 题），从 `Redis.md` 知识文档中提取深度技术问题，覆盖 I/O 多路复用、跳表结构、渐进式 Rehash、分布式锁、集群分片等全部核心知识点，用来严格评估 RAG 管道的检索精度。

脚本用 `requests.post` 向 `/api/chat` 端点顺序发 JSON 请求（`{prompt, sessionId, userId}`），题与题之间休眠 2 秒，防止触发阿里云 API 的并发限流（HTTP 429）。脚本同时支持向 V1（旧版本）和 V2（新版本）两个独立实例并行发请求，方便做 A/B 对比实验。

**第二层——指标采集。** 后端 Spring Boot 应用通过 `spring-boot-starter-actuator` 加 `micrometer-registry-prometheus` 把系统运行指标暴露在 `/api/actuator/prometheus` 端点上。RAG 检索相关的核心指标包括 `rag_retrieval_hit_total`（命中次数）、`rag_retrieval_miss_total`（未命中次数）、`rag_retrieval_duration_seconds`（检索耗时分布）。大模型调用侧的指标（请求总数、成功率、Token 消耗、响应延迟分位数）通过 `AiModelMetricsCollector` 与 `AiModelMonitorListener` 采集，流式场景下的首字响应时间（TTFT）在 `AiChatController.streamChat()` 里用 `AtomicBoolean` 捕获。

**第三层——可视化。** Prometheus 以 Pull 模式定时抓取指标端点，Grafana 通过 PromQL 查询将时序数据渲染为仪表盘面板（`grafana.json`）。仪表盘包括顶部统计区（总请求数、真实成功率、平均响应时间、错误总数）、时序分析区（请求状态分布、P50/P90/P99 延迟分位数、Token 消耗趋势）、用户与会话排名区（调用频次 Top 5、Token 消耗 Top 5），以及按 `model_name` 标签切换监控目标的变量选择器。

#### 3.3.3 核心评测指标：Hit@K

在所有监控指标中，**Hit@K**（前 K 条检索结果中包含正确答案的比例）是驱动 RAG 管道持续演进的核心指标。其定义为：对于测试集中的每道知识型问题，若该题的标准答案所在文档片段出现在 RAG 管道最终返回的 Top-K 条结果中，则记为该题命中；所有题目的命中数与总题数的比值即为 Hit@K。

Grafana 仪表盘在最终版本中独立呈现三个粒度的 Hit@K 面板：

| 指标      | 含义                          | 关注焦点                                                             |
| --------- | ----------------------------- | -------------------------------------------------------------------- |
| **Hit@1** | 首条结果即包含正确答案的比例  | 精排模型的语义对齐精度——衡量 Rerank 阶段将最相关文档置于首位的准确率 |
| **Hit@3** | 前 3 条结果包含正确答案的比例 | 粗排召回率与精排排序质量的综合体现——Top-3 足以覆盖大多数单轮问答场景 |
| **Hit@5** | 前 5 条结果包含正确答案的比例 | RAG 管道的整体覆盖率上限——反映粗排阶段候选集是否充分覆盖知识库       |

典型的优化决策流程是：若 Hit@5 偏低（< 85%），优先排查粗排阶段——调整 `maxResults` 增加候选池尺寸、降低 `minScore` 放宽准入门槛、优化 Embedding 模型选型或查询预处理逻辑；若 Hit@5 较高但 Hit@1 偏低（< 60%），则重点调优精排阶段——升级 Rerank 模型、调整 `finalTopN` 截断策略、优化 Prompt 中知识片段的注入方式。通过分指标定位、分阶段优化，每次变更的效果都可以被精确量化，避免了盲目调参。

#### 3.3.4 RAG 优化闭环

本系统所有 RAG 相关的技术选型与参数配置均经历了"基线测量 → 单变量变更 → A/B 对比 → 指标确认"的完整闭环。具体实践包括：

- **Embedding 模型选型**：以 Hit@1/Hit@3/Hit@5 为评价标准，对比不同 Embedding 模型（text-embedding-v2 vs text-embedding-v4）在相同测试集上的检索表现，选定 text-embedding-v4 基于其 1024 维向量在中文技术文档场景下的语义区分能力。
- **粗排参数调优**：在固定精排模型的前提下，以 Hit@5 为优化目标，逐次调整 `maxResults`（20→25→30→35）和 `minScore`（0.65→0.60→0.55），选取 Hit@5 达到 90% 以上且检索耗时可控的参数组合。
- **Prompt 模板迭代**：系统 Prompt（`system-prompt/chat-bot.txt`）中嵌入的问题改写策略与 Few-shot 示例，均通过对比实验验证了其在提升 RAG 检索命中率方面的有效性——改写后的查询语句能更好地匹配知识库文档的术语体系。
- **文档切分策略**：`RecursiveDocumentSplitter` 的 `maxChunkSize`（800）与 `chunkOverlap`（200）参数经过多轮测试选定，以 Hit@K 为评价标准，平衡了语义完整性与检索粒度。

通过这套方法，系统的每一次改动都有数据可查、可复现、可量化，而不是靠个人经验或社区流行度来判断好坏。

### 3.4 系统部署

#### 3.4.1 部署拓扑

系统的基础设施层依赖两个有状态中间件——PostgreSQL（含 PgVector 插件）及 Redis，推荐与无状态应用服务共同编排在 Docker 容器化环境中。整体部署拓扑如下：

```

                    │   Nginx / LB     │
                    │   (反向代理)      │
                    └───────┬──────────┘
                            │
              ┌─────────────┼─────────────┐
              ▼             ▼             ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ App-1    │ │ App-2    │ │ App-N    │
        │ :10010   │ │ :10010   │ │ :10010   │
        │ (Spring  │ │ (Spring  │ │ (Spring  │
        │  Boot)   │ │  Boot)   │ │  Boot)   │
        └────┬─────┘ └────┬─────┘ └────┬─────┘
             │            │            │
             └────────────┼────────────┘
                          │
              ┌───────────┼───────────┐
              ▼                       ▼
        ┌──────────┐           ┌──────────────┐
        │  Redis   │           │ PostgreSQL    │
        │  :6379   │           │ + PgVector    │
        │  (会话    │           │ :54328        │
```

**应用服务（Spring Boot Jar）。** 应用本身为无状态服务，所有会话状态外置至 Redis，所有向量数据外置至 PostgreSQL。这一设计支持水平扩展——实例数量可根据负载弹性伸缩，同一会话的多次请求即使被负载均衡器分发至不同实例，由 MemoryId 索引的 Redis 记忆存储保证上下文一致性。

**PostgreSQL + PgVector。** `PgVectorEmbeddingStore` 配置中 `dimension=1024` 与 text-embedding-v4 模型的输出向量维度对齐。表结构通过 `createTable=true` 自动创建，`dropTableFirst=true` 确保每次重启时清空旧向量数据并重新加载文档（适用于开发阶段；生产环境应设为 `false`）。

**Redis。** `RedisChatMemoryStore` 通过 `RedisChatMemoryStoreConfig` 装配，构建参数包括 host、port、password、user 和 ttl。`ttl=3600` 秒的过期策略使得 1 小时无活动的会话记忆自动清理，平衡了用户体验与存储成本。

#### 3.4.2 环境隔离策略

系统采用 Spring Profiles 机制实现多环境配置隔离：

- **`application.yml`（默认配置）**：包含通用结构声明与生产环境实际连接信息。模板中 API Key 字段以占位符 `xxxxxx` 标注，实际部署时通过环境变量覆盖（`SPRING_APPLICATION_JSON` 或 `-D` JVM 参数）或外部化配置中心注入。
- **`application-local.yml`（本地开发配置）**：覆盖默认配置中的敏感字段，将 Redis 与 PostgreSQL 地址指向本地 Docker 容器（`127.0.0.1`），使用本地专用 API Key。通过 `spring.profiles.active=local` 激活。

该策略的隔离粒度精确到单个配置属性级别——本地环境仅需覆盖 `host`、`password`、`api-key` 等敏感字段，而 `rag.docs-path`、`chat.memory.compression.*`、`server.port` 等通用配置由默认文件统一管理，避免了配置冗余。

#### 3.4.3 可观测性基础设施

系统通过 `spring-boot-starter-actuator` + `micrometer-registry-prometheus` 暴露监控端点。配置中 `management.endpoints.web.exposure.include=health,info,prometheus`（生产环境）或 `"*"`（本地环境）将 Prometheus 指标暴露于 `/api/actuator/prometheus`。Grafana 通过该端点拉取指标，构建包括 Token 消耗趋势图、模型调用 QPS、RAG 命中率曲线、用户会话排行榜在内的运维仪表盘。

## 五、项目特色之处

### 5.1 解耦的多 Agent 协同架构

当前业界不少基于 LangChain 的"Agent"项目实际上只是对单次大模型调用的薄封装——所有逻辑挤在 Prompt 模板里，缺乏结构化的任务分解。本系统通过 `Agent` 接口抽象、`KnowledgeAgent`/`ReasoningAgent` 的职责分离和 `SimpleOrchestrator` 的规则路由，把"找资料"和"用资料生成回答"拆成了两个独立模块。两个 Agent 通过增强输入字符串这个简单契约协作。以后的扩展也比较方便——新增一个 `Agent` 实现类并在编排器里加条路由规则，就能接入新的专业 Agent（如代码执行 Agent、数据库查询 Agent），不用改现有代码。

### 5.2 两阶段检索解决 RAG 粗排精度不足

纯向量检索的 RAG 系统存在一个老问题：双塔模型把查询和文档分别编码成独立向量，两者之间没有细粒度的语义交互，Top-K 结果里容易混进"看着相关其实没用"的噪声文档。本系统用"向量粗排 + 交叉编码器精排"两阶段来解决：第一阶段从海量文档中召回 30 条候选，优先保召回率；第二阶段用 qwen3-rerank 交叉编码器模型对候选文档逐条打分，最终把语义最对齐的 5 条文档送入大模型上下文。整条链路有降级容错兜底，重排序服务出故障时系统还能正常运行。

### 5.3 基于真实 Token 计数的自适应记忆压缩

大模型 API 有两个硬约束：按 Token 收费，以及上下文窗口有上限。在需要长期陪跑用户的技术辅导场景，对话可能累积到上百轮，不控制的话费用会飙升甚至超出窗口限制。`CompressibleChatMemory` 通过三项设计来解决：

- **精确 Token 计量**：不用简单的字符或单词估算，而是用 HuggingFace Tokenizers 加载 Qwen 原生词表做真实 Token 计数，确保阈值判断和实际 API 计费口径一致。
- **双维度触发**：同时看消息数量（>20 条）和 Token 总量（>6000），任意一个超了就触发压缩，短消息高频对话和长文本粘贴两种情况都能覆盖。
- **安全降级**：三层保护——压缩失败就简单截断保留最后 10 轮、加锁失败就降为非原子写入、近期消息 Token 超限只告警不丢数据。

系统在压缩同时引入了 Redis 分布式锁保护并发安全，这套机制可以在多实例部署下可靠运行。相比调大模型做摘要压缩，当前用轻量级文本截取生成摘要的方案在延迟和成本上都更有优势。

### 5.4 全链路可观测性覆盖大模型调用

大模型应用的运行特征和传统 CRUD 系统差别很大——调用延迟是秒级的、强依赖外部 API 可用性、Token 消耗直接关联运营成本。本系统从 HTTP 请求到 LLM 响应拉了一条完整的可观测链路：

- **请求级追踪**：`ObservabilityInterceptor` 在 HTTP 层注入 `X-Request-ID`，通过 `MonitorContextHolder`（基于 `InheritableThreadLocal`）传递 `{requestId, userId, sessionId}`，确保同一请求链路上的日志和指标能关联起来。
- **LLM 调用监听**：`AiModelMonitorListener` 实现了 LangChain4j 的 `ChatModelListener` 接口，在 `onRequest`/`onResponse`/`onError` 三个生命周期钩子里采集延迟、Token 用量（输入/输出/总计）和错误信息，通过 `AiModelMetricsCollector` 以多维度标签（user_id、session_id、model_name、status）注册到 Micrometer。
- **流式延迟指标**：在流式对话场景里，`AiChatController.streamChat()` 用 `AtomicBoolean` 捕获首字响应时间（TTFT），在 `doOnComplete` 回调里计算生成耗时和总耗时，弥补了标准监听器在流式场景下不好区分首字延迟和生成延迟的问题。
- **RAG 专项指标**：`RagMetricsCollector` 单独采集检索命中/未命中次数和检索耗时，为 RAG 管道的持续调优提供数据支持。

## 六、问题与解决对策

### 6.1 LangChain4j 版本 API 兼容问题

**现象**：LangChain4j 从 beta 版本升级至 1.1.0 正式版后，`EmbeddingStore` 的检索 API 签名发生变化——旧版的 `search(Embedding, int, double)` 方法被替换为基于 `EmbeddingSearchRequest` 构建器的新 API。编译时报方法签名不匹配错误。

**对策**：查阅 LangChain4j 1.1.0 的官方迁移文档，将 `RagTool.retrieve()` 和 `ReRankingContentRetriever` 中的检索调用全部迁移至新 API：通过 `EmbeddingSearchRequest.builder().queryEmbedding(embedding).maxResults(k).minScore(threshold).build()` 构造请求，再调用 `embeddingStore.search(request).matches()` 获取结果。同时修复了元数据读取方式——1.1.0 中 `Metadata.getString()` 在 key 不存在时可能抛异常，改为先判断返回值是否为 null 再使用。

**反思**：框架升级前应先建一个独立分支跑一遍完整的编译和测试流程，确认所有 API 变更点后再合并。这次在主分支上直接升级导致了一段短暂的编译不可用窗口。

### 6.2 PgVector 向量维度不匹配

**现象**：更换 Embedding 模型从 text-embedding-v2 到 text-embedding-v4 后，v4 模型输出 1024 维向量，但 PgVector 建表时仍沿用旧版的维度参数，导致向量插入时报 "expected 1536 dimensions, got 1024" 错误。

**对策**：在 `EmbeddingStoreConfig` 中将 `dimension` 参数由旧值更新为 `1024`，并设置 `dropTableFirst=true` 重建表结构。正式上线前将 `dropTableFirst` 改回 `false` 以避免生产环境丢数据。

**反思**：Embedding 模型和向量数据库的维度约束是隐式耦合点，应该在配置类中加上启动期校验——读取 Embedding 模型的实际输出维度，与 PgVector 的配置维度做比对，不匹配则拒绝启动并给出明确报错信息。

### 6.3 JAR 包内嵌资源无法直接读取

**现象**：在 IDE 中直接运行时，`ClassPathResource("qwen/tokenizer.json")` 能正常通过文件路径读取。但打成 Spring Boot fat JAR 后，类路径资源变成了 JAR 内嵌条目，无法通过 `Path` 直接引用，`HuggingFaceTokenizer.newInstance(Path)` 抛 `FileNotFoundException`。

**对策**：在 `ChatMemoryConfig.qwenTokenizer()` 中改用"类路径流复制到临时文件"的方式：`InputStream is = resource.getInputStream()` → `Files.copy(is, tempFile, StandardCopyOption.REPLACE_EXISTING)` → `HuggingFaceTokenizer.newInstance(tempFile)`，并在 JVM 退出时删除临时文件（`tempFile.toFile().deleteOnExit()`）。

**反思**：框架和工具库对"资源"的抽象不一致（Spring 的 `Resource` vs DJL 的 `Path`），集成时需要特别注意边界转换。这个问题的排查花了不少时间，因为 IDE 中一切正常，只在部署阶段暴露，属于典型的"环境差异 bug"。

### 6.4 阿里云 API 并发限流（HTTP 429）

**现象**：高频率连续调用 Qwen API 时，后端日志出现大量 `429 Too Many Requests` 错误，尤其在压测脚本连续发送 50 道题时最为明显。错误重试进一步加剧了 Rate Limit 的消耗。

**对策**：(1) 测试脚本中每题之间添加 `time.sleep(2)` 强制间隔；(2) 在 `AiModelMonitorListener.onError()` 中识别 429 错误并单独记录，不与其他业务错误混淆；(3) 后续计划引入令牌桶限流器在应用层做本地流控，从源头上控制请求发射速率。

**反思**：依赖第三方 API 的系统必须做两层防护——应用层的本地限流 + API 侧的 HTTP 状态码识别与退避重试。单纯依赖客户端的 sleep 不够稳健，应该在服务端也加一层控制。

### 6.5 敏感词误拦截技术术语

**现象**：技术问答中包含的某些英文缩写、变量命名或数据库命令被敏感词库误识别为违禁词，导致正常的后端技术提问被拦截，用户反馈"问个 Redis 命令都发不出去"。

**对策**：(1) 在 `SensitiveWordConfig` 中关闭数字校验（`enableNumCheck(false)`）、邮箱校验（`enableEmailCheck(false)`）和 URL 校验（`enableUrlCheck(false)`），避免数字串、邮箱地址和网址被误判；(2) 利用 `CustomWordAllow` 白名单机制，将确认安全的技术术语加入 `sensitive-word-allow.txt`，每次部署前可更新该文件。

**反思**：敏感词库的初始默认策略偏向严格，在技术问答场景下需要做针对性宽松调整。白名单文件的设计在这一点上体现了灵活性——不需要改代码，改完配置文件重启即生效。
